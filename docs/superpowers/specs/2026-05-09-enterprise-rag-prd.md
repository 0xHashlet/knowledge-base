# 权限感知企业知识库 RAG 平台 — 产品需求文档

## 1. 产品定位与愿景

### 1.1 产品定义

**权限感知企业知识库 RAG 平台**是一套私有化部署的智能问答系统。它将企业内部文档（制度、合同、产品文档、技术资料等）转化为可检索、可对话的知识资产，员工用自然语言提问，系统基于文档内容生成精准回答并附原文溯源。

### 1.2 核心价值主张

| 维度 | 描述 |
|------|------|
| **安全性** | 用户无权访问的文档内容，不进入检索召回、不进入 LLM 上下文、不出现在回答中 |
| **准确性** | 回答基于企业真实文档，每条论断附溯源引用，可追溯不可编造 |
| **私有化** | 数据全链路不出企业内网，LLM 默认本地部署，适配企业安全合规要求 |
| **可演进** | 结构化评测体系驱动质量迭代，badcase 可定位、可复现、可修复 |

### 1.3 不做什么

- 不做公有 SaaS 多租户版本
- 不做通用搜索引擎（只检索企业已上传的文档）
- 不做知识库协作编辑（文档创作在平台外完成，平台只管理已成型文档）
- 不做实时数据源接入（数据库、API 实时查询），专注文档型知识

## 2. 用户角色与核心场景

### 2.1 角色总览

| 层级 | 角色 | 核心职责 |
|------|------|----------|
| 内容消费者 | 普通员工 | 在权限内提问，获取答案与溯源 |
| 内容贡献者 | 知识库编辑 | 上传/管理文档，维护内容质量 |
| 内容贡献者 | 知识库负责人 | 管理知识库配置、成员与健康度 |
| 组织管理者 | 部门管理员 | 管理部门用户与知识库访问范围 |
| 组织管理者 | 系统管理员 | 全局用户/角色/系统配置/审计 |
| 质量与治理 | 评测标注人员 | 标注评测集，追踪 badcase |
| 质量与治理 | 合规审计人员 | 审查访问日志，验证权限有效性 |

### 2.2 核心用户旅程

**普通员工 — "找一个答案"**

```text
登录 → 选择知识库范围 → 输入自然语言问题
→ 系统检索召回 + LLM 生成回答（附引用标注）
→ 查看原文溯源（仅可见有权文档的片段）
→ 对回答提交反馈（有用/无用/不准确）
→ 可基于前文继续追问
```

**知识库编辑 — "传一篇文档，让它能被搜到"**

```text
登录 → 进入所管理的知识库 → 上传文档（支持批量）
→ 系统异步解析 + chunk 生成 + embedding 写入
→ 查看解析状态（成功/处理中/失败）
→ 对失败文档查看错误原因并重试
→ 必要时下架或替换文档版本
```

**知识库负责人 — "我的知识库健康吗"**

```text
登录 → 查看知识库概览面板
→ 文档数量/状态分布、高频提问 Top N、低分回答列表
→ 发现某类问题未覆盖 → 安排编辑补充文档
→ 发现某篇文档已过时 → 归档并上传新版本
→ 管理成员：添加编辑者、调整权限
```

**评测标注人员 — "这个回答准不准"**

```text
登录 → 进入评测工作台
→ 查看待标注问答对列表
→ 对每个回答标注：准确性/完整性/相关性打分 + 问题分类
→ 标记 badcase 并归类原因（检索遗漏/LLM 错误/权限过滤过度等）
→ 管理评测数据集，触发回归评测
```

## 3. 功能规格

### 3.1 认证与权限体系

#### 3.1.1 认证

- **本地账号**：用户名 + 密码，JWT 令牌认证，支持 token 刷新
- **企业 SSO**（L1）：OIDC 身份联邦，支持配置为主要或辅助认证方式
- **LDAP/AD**（L2）：企业域控认证
- **第三方 OAuth**（L2）：钉钉/飞书/企微快捷登录
- **会话管理**：JWT 有过期时间，支持主动登出，管理员可强制下线特定用户

#### 3.1.2 权限模型

采用 **RBAC（基于角色的访问控制）** + **资源级授权**：

```
用户 ──(绑定)──▶ 角色 ──(绑定)──▶ 权限
                       │
用户 ──(直接)──▶ 知识库成员关系（Owner / Editor / Viewer）
```

- **角色**：系统预置角色 + 可自定义角色
- **权限**：细粒度权限点，如 `kb:read`、`kb:write`、`user:manage`、`system:config`
- **知识库成员关系**：独立于全局角色的资源级授权，用户在知识库中的角色决定对该知识库内文档的访问级别
- **权限判定优先级**：系统管理员 > 知识库 Owner > 知识库 Editor > 知识库 Viewer > 普通用户

#### 3.1.3 文档级权限过滤

- 知识库成员关系决定用户对知识库内文档的可见性
- 检索召回前：过滤掉用户无权访问的文档 ID，后续召回仅在有权限的文档范围内进行
- 问答回答中：溯源引用仅展示有权文档的片段，无权限文档不出现在任何输出中
- 单一知识库内可对单篇文档设置例外访问规则（L2）

#### 3.1.4 数据隔离

| 层级 | 实现方式 |
|------|----------|
| 知识库成员隔离 | 非成员无法访问知识库的任何内容 |
| 文档级过滤 | 检索前按文档权限过滤候选集 |
| 溯源裁剪 | 回答引用中仅展示有权文档片段 |
| 审计留痕 | 所有提问和检索行为记录到审计日志 |

### 3.2 知识库管理

#### 3.2.1 知识库生命周期

```text
创建 → 配置 → 运营 → 归档
```

- **创建**：设定名称、描述、可见范围（全组织 / 指定部门）
- **配置**：成员管理、默认 LLM 选择、文档审批策略（L2）
- **运营**：日常文档上传、检索问答、健康度监控
- **归档**：不再使用的知识库设为只读归档，不可新增文档和提问，可恢复

#### 3.2.2 知识库成员管理

- 三种成员角色：Viewer（只读提问）、Editor（可上传文档）、Owner（完全管理）
- Owner 可添加/移除成员、变更成员角色
- 支持从部门批量导入成员（L2）
- 成员权限变更即时生效，不依赖 token 刷新

#### 3.2.3 知识库元数据

| 字段 | 说明 |
|------|------|
| 名称 | 必填，唯一 |
| 描述 | 可选，富文本 |
| 可见范围 | 全组织 / 指定部门 |
| 状态 | Active / Archived |
| 创建者/负责人 | 关联用户 |
| 文档数量 | 统计值 |
| 创建/更新时间 | 自动维护 |

#### 3.2.4 健康度概览（L2）

知识库负责人视角面板：

- 文档状态分布（已就绪 / 解析中 / 解析失败）
- 近 7/30 天提问量趋势
- 高频未覆盖问题（提问多但无匹配文档）
- 低分回答 Top 10（需补充或修正的文档）

### 3.3 文档全生命周期

#### 3.3.1 文档上传

- 支持格式：text/plain、PDF、Word (.docx)、Excel (.xlsx)、PPT (.pptx)、Markdown
- 上传方式：单文件上传、批量上传（L2）
- 上传校验：文件类型白名单、单文件大小限制（可配置，默认 50MB）
- 上传时校验：当前用户对该知识库的写权限
- 同名文件不覆盖，生成新版本

#### 3.3.2 文档版本管理

```text
文档记录（documents）1 : N 文档版本（document_versions）
```

- 每次上传创建新版本，保留完整历史
- 版本状态流转：`PENDING → PROCESSING → READY / FAILED`
- 仅 READY 状态的版本参与检索召回
- 支持版本回滚：将指定历史版本重新激活为当前活跃版本
- 旧版本不删除，标记为 superseded

#### 3.3.3 文档解析管道

```text
上传 → 类型检测 → 文本提取 → chunk 切分 → embedding → 写入 Milvus
```

**文本提取**：按文件类型路由到对应解析器

| 格式 | 解析器 | 策略 |
|------|--------|------|
| PDF | PyMuPDF | 提取文本，保留页码和章节标题 |
| Word (.docx) | python-docx | 提取结构化文本，保留标题层级 |
| Excel (.xlsx) | openpyxl | 按工作表逐行提取，每行生成为 `列名: 值` 结构化文本，合并单元格展开填充 |
| PPT (.pptx) | python-pptx | 按幻灯片提取文本，保留标题与正文层级 |
| Markdown / 纯文本 | 直接读取 | 保留章节标题结构 |

**chunk 切分**：可配置策略（固定长度 / 按段落 / 按章节标题），默认按段落 + 重叠窗口（重叠 50 字符）

**embedding 生成**：

- L1 默认：BAAI `bge-large-zh-v1.5`，本地部署，维度 1024，HuggingFace Transformers / TEI 加载
- 接口抽象，可替换为其他模型或外部 API
- embedding 服务独立部署，API 通过 OpenAI-compatible 接口调用

**写入向量库**：chunk embedding 写入 Milvus，chunk 文本和元数据写入 PostgreSQL

**失败处理**：解析异常捕获详细错误信息，标记 FAILED，支持手动重试

#### 3.3.4 文档状态管理

| 状态 | 说明 | 参与检索 |
|------|------|:---:|
| PENDING | 已上传，等待解析 | ❌ |
| PROCESSING | 解析进行中 | ❌ |
| READY | 解析完成，可检索 | ✅ |
| FAILED | 解析失败，需人工处理 | ❌ |
| ARCHIVED | 已归档，历史版本 | ❌ |

- 编辑可手动将文档标记为 ARCHIVED（下架）
- ARCHIVED 文档不参与检索，但保留历史回答中的溯源引用

### 3.4 检索与问答引擎

#### 3.4.1 检索链路

```text
用户提问 → 权限过滤 → 关键词召回 → 向量召回 → 合并去重 → rerank → 上下文构造 → LLM 生成 → 溯源标注 → 回答返回
```

| 步骤 | 输入 | 输出 | 说明 |
|------|------|------|------|
| 权限过滤 | 用户 ID、知识库范围 | 有权文档 ID 列表 | 召回前剪枝，无权文档不进入后续 |
| 关键词召回 | 问题文本 + 文档 ID 列表 | chunk 候选集 A | BM25 全文检索，保证精确术语命中 |
| 向量召回 | 问题 embedding + 文档 ID 列表 | chunk 候选集 B | Milvus ANN 检索，覆盖语义近似匹配 |
| 合并去重 | 候选集 A + B | 去重后候选集 | 按 chunk ID 去重，保留来源信息 |
| rerank | 去重候选集 + 问题 | 排序后 Top-K chunk | 重排序模型，精确匹配度排序 |
| 上下文构造 | Top-K chunk + 问题 | prompt | 拼接 chunk 文本 + system prompt + 用户问题，使用 ChatML 格式 |
| LLM 生成 | prompt | 回答文本 | 本地模型推理 |
| 溯源标注 | 回答 + 使用的 chunk | 回答 + 引用标记 | 标注每句话对应的源文档和片段 |
| 回答返回 | 回答 + 引用 | API 响应 | 含溯源列表、置信度（L2） |

**rerank 模型**：

- L1 默认：BAAI `bge-reranker-v2-m3`，本地部署，多语言，MIT 开源协议可商用
- 通过 HuggingFace Transformers 加载，或从 ModelScope 下载
- 接口抽象，可替换

**关键词召回**：

- L1：BM25（elasticsearch 或本地 BM25 实现），替代当前 ILIKE 方案
- 中文分词使用 jieba

#### 3.4.2 多轮对话

- 对话上下文窗口：最近 N 轮（可配置，默认 5 轮）
- 每轮记录用户问题和系统回答
- 追问时上下文窗口内的历史 Q&A 一并送入 prompt
- 切换知识库范围时清空上下文
- 上下文窗口溢出时 FIFO 淘汰最早轮次
- **存储方案**：对话历史存储于 Redis，key 格式 `chat:{session_id}`，TTL 与 JWT 过期时间一致

#### 3.4.3 引用溯源

回答中每条事实性论断标注源文档：

```json
{
  "answer": "根据《员工手册》第三章，年假天数为...",
  "citations": [
    {
      "document_id": "doc_123",
      "document_title": "员工手册-2025版",
      "version_id": "ver_456",
      "chunk_id": "chunk_789",
      "chunk_text": "员工每年享有带薪年假...",
      "relevance_score": 0.92
    }
  ]
}
```

- 前端可点击引用跳转到原文片段
- 引用原文片段受文档权限控制——用户无权限的文档片段不展示原文，仅标记为"受权限限制"

#### 3.4.4 LLM 适配层

```text
LLM 抽象接口
  ├── 本地模型适配器（vLLM / Ollama / 其他 OpenAI-compatible 服务）
  ├── 外部 API 适配器（OpenAI / 通义千问 / DeepSeek 等，作为可选能力）
  └── 模型路由（按知识库配置选择模型，L2）
```

- 统一接口：`generate(prompt, model_config) → response`
- 模型配置：endpoint、model name、temperature、max_tokens、系统 prompt 模板
- 全局默认模型 + 知识库可覆盖
- 调用日志：记录每次 LLM 调用的模型、延迟、token 消耗（L2）
- fallback 链（L2）：主模型不可用时自动切换到备用模型

#### 3.4.5 安全守卫

- **系统 prompt 注入防护**：用户输入与系统指令使用 ChatML 格式严格隔离，系统指令在 `<system>` 块内，用户输入先经过格式化转义再放入 `<user>` 块
- **回答安全校验**（L2）：敏感信息检测（手机号、身份证号等 PII），避免 LLM 生成虚构信息时输出格式化的假数据
- **检索无结果处理**：候选集为空时直接告知用户"未找到相关文档"，不进入 LLM 生成（避免模型编造）
- **权限强制兜底**：LLM 生成完成后，二次校验回答中引用的 chunk 是否在用户有权范围内，防止链路中任何环节绕过权限

### 3.5 评测与质量体系

#### 3.5.1 用户反馈

- 每条回答下提供快捷反馈：有用 / 无用 / 不准确
- 无用或不准确时可附文字说明
- 反馈关联到具体问答记录，记录提问、回答、使用的 chunk、用户 ID
- 反馈统计面板（L2）：按知识库、时间段查看反馈分布

#### 3.5.2 评测数据集

- 评测数据集 = 一组问答对（问题 + 期望答案 + 期望引用文档）
- 支持手动创建和从用户反馈中导入
- 数据集按知识库关联，可打标签分类（准确性 / 完整性 / 安全性）
- 黄金测试集：标记为回归测试的固定数据集，每次变更后必跑

#### 3.5.3 评测执行

```text
评测数据集 → 逐条执行检索+生成 → 对比期望结果 → 输出评测报告
```

- 触发方式：手动触发 / CI 集成触发 / 定时任务
- 评测指标：
  - **Hit Rate@K**：期望文档是否进入召回 Top-K（衡量检索召回能力）
  - **MRR（Mean Reciprocal Rank）**：期望文档在召回结果中的排名倒数平均（衡量排序质量）
  - **NDCG@K**（L2）：归一化折损累计增益（衡量多相关文档的排序质量）
  - **Faithfulness**（L2）：LLM 评判回答中的事实是否忠于源文档
  - **Answer Relevancy**（L2）：LLM 评判回答与问题的相关性
- 评测结果存储，支持历史对比（本次 vs 上次）

#### 3.5.4 Badcase 追踪

- Badcase 来源：用户低分反馈、评测标注发现的低质量回答
- Badcase 分类：检索遗漏 / LLM 错误 / 权限过滤过度 / 文档质量问题 / prompt 问题
- 每条 badcase 有状态：待处理 → 分析中 → 已修复 → 已验证
- 修复后回归验证：badcase 关联到评测数据集中的条目，修复后重跑确认

#### 3.5.5 评测工作台（L2）

评测标注人员专用界面：

- 待标注列表：展示待评测问答对
- 标注表单：准确性 1-5 / 完整性 1-5 / 相关性 1-5 + 问题分类
- 批量操作：筛选、批量标注、导出
- 标注进度统计

### 3.6 系统集成接口

#### 3.6.1 企业 SSO 集成

| 协议 | 说明 | 优先级 |
|------|------|:---:|
| OIDC | 基于 OpenID Connect 的标准身份联邦 | L1 |
| LDAP/AD | 企业域控认证 | L2 |
| OAuth 2.0 | 可接入钉钉/飞书/企微等第三方登录 | L2 |

- 支持多身份源配置，本地账号与 SSO 账号可并存
- SSO 用户首次登录自动创建本地用户记录
- 属性映射：SSO 返回的部门/组信息可映射为平台内的部门和角色

#### 3.6.2 消息通知集成

| 渠道 | 说明 | 优先级 |
|------|------|:---:|
| 邮件 | SMTP 发送，文档解析完成/失败通知 | L1 |
| 钉钉 | 机器人 Webhook | L2 |
| 飞书 | 机器人 Webhook | L2 |
| 企业微信 | 机器人 Webhook | L2 |

- 通知事件：文档解析完成/失败、评测报告生成、知识库异常告警
- 用户可在个人设置中选择通知偏好（L2）

#### 3.6.3 审计系统对接

- 审计日志格式：结构化 JSON，含时间、用户、操作、资源、结果
- 对接方式：Syslog 转发 / 写入企业日志中心 / Kafka 推送（L2）
- 本地保留完整审计日志，外部对接为增量输出

#### 3.6.4 对象存储适配

```text
对象存储抽象层
  ├── MinIO（开发/小型部署）
  ├── 阿里云 OSS
  ├── AWS S3
  └── 腾讯 COS
```

- 统一 S3-compatible 接口抽象
- 数据库仅存 object key，不存完整 URL
- 存储切换只需改配置 + 数据迁移，不涉及代码变更

#### 3.6.5 REST API 与 SDK

- 所有后端能力通过 RESTful API 暴露
- OpenAPI 3.0 文档自动生成（FastAPI 内置，`/docs` 可交互调试）
- L2：提供 Python SDK 封装常用 API 调用
- API 版本策略：URL 路径版本 `/api/v1/`，破坏性变更走新版本，旧版本保留 2 个大版本周期

- Alembic 数据库迁移，版本化管理
- 向后兼容：新的必填列必须有默认值
- 灰度发布支持（L2）：可先升级部分 worker 节点

## 4. 非功能需求

### 4.1 安全

| 需求 | 说明 | 优先级 |
|------|------|:---:|
| 传输加密 | 全链路 HTTPS/TLS | L1 |
| 密码安全 | bcrypt 哈希存储，强度可配置 | L1 |
| JWT 安全 | 签名验证、过期策略、secret 轮换 | L1 |
| 权限强制执行 | 权限过滤在检索链路最先执行，LLM 生成后二次校验，不可绕过 | L1 |
| 日志脱敏 | 日志中的敏感信息（密码、token）自动脱敏 | L1 |
| PII 检测 | 回答中的手机号、身份证号等敏感信息检测 | L2 |
| API 限流 | 按用户/IP 限流，防暴力破解和滥用 | L2 |
| 依赖安全 | 定期扫描第三方依赖漏洞 | L2 |
| 数据加密 | 对象存储中文件支持服务端加密（SSE） | L2 |
| 安全审计 | 所有敏感操作留痕，不可篡改 | L1 |

### 4.2 性能

| 指标 | 目标值 | 说明 |
|------|--------|------|
| API 响应时间（健康检查） | < 50ms | 无负载 |
| 检索延迟（关键词+向量） | < 500ms | 不含 LLM 生成 |
| 端到端问答延迟 | < 5s | 提问到回答返回完整链路 |
| 文档解析吞吐 | > 10 页/秒 | PDF 单文件解析 |
| 并发提问 | 50 并发 | 基准值，随部署规模可扩展 |
| LLM 推理延迟 | 取决于模型和硬件 | 本地模型，不做硬性承诺 |

### 4.3 可用性

| 需求 | 说明 |
|------|------|
| 服务可用性 | 核心问答链路 99.5%（允许计划内维护窗口） |
| 优雅降级 | LLM 不可用时返回"服务暂不可用"，不输出无 LLM 的错误回答；Milvus 不可用时关键词召回可独立工作 |
| 故障恢复 | PostgreSQL/Milvus/Redis 使用 Docker 自动重启策略 |

### 4.4 可扩展性

| 维度 | 策略 |
|------|------|
| 文档量 | PostgreSQL 全文检索 + Milvus 向量检索，均可水平扩展 |
| 用户量 | 无状态 API 实例可横向扩展，Celery worker 可独立扩展 |
| 模型切换 | LLM 抽象层统一接口，新增模型仅需加适配器 |
| 存储 | 对象存储抽象，S3-compatible 即可接入 |
| 向量维度 | embedding 模型维度可配置，切换模型时重建 collection |
| embedding 模型 | 抽象接口，支持本地模型和外部 API 两种模式 |

### 4.5 可维护性

| 需求 | 说明 |
|------|------|
| 结构化日志 | JSON 格式，含 trace_id 串联完整请求链路 |
| 配置管理 | 环境变量 + `.env` 文件，不硬编码 |
| 数据库迁移 | Alembic 管理所有 schema 变更，可回滚可追溯 |
| API 文档 | OpenAPI 3.0 自动生成，`/docs` 可交互调试 |
| 代码规范 | 分层清晰（API → Service → Repository），每层职责明确 |

### 4.6 兼容性

| 维度 | 要求 |
|------|------|
| 浏览器 | Chrome/Edge/Firefox 最近 2 个大版本 |
| Python | 3.12+ |
| 数据库 | PostgreSQL 16 |
| 向量库 | Milvus 2.5.x |
| 容器运行时 | Docker / Docker Compose，可迁移到 K8s（L2） |

## 5. 交付路线图

### L1 — 核心可用（目标：可上线试用）

最小可用版本：一个员工能从企业内部知识库中问到准确答案。

| 功能域 | 范围 |
|------|------|
| 认证 | 本地账号 + JWT + OIDC SSO |
| 权限 | RBAC + 知识库成员关系 + 文档级权限过滤 |
| 知识库 | CRUD + 成员管理 |
| 文档 | 上传（text/PDF/Word/Excel/PPT/Markdown）、解析、chunk、embedding（bge-large-zh-v1.5）、版本管理 |
| 检索 | 权限过滤 → BM25 关键词召回 → 向量召回 → 合并去重 → rerank（bge-reranker-v2-m3） → LLM 生成 → 溯源标注 |
| 问答 | 单轮对话 + 多轮追问（对话历史存 Redis），引用溯源 |
| LLM | 本地模型（vLLM/Ollama，OpenAI-compatible 接口），单模型，无 fallback |
| 管理台 | 用户门户（问答页）+ 管理后台（知识库、文档、成员管理）+ 系统设置 |
| 反馈 | 有用/无用/不准确快捷反馈 |
| 集成 | OIDC SSO、SMTP 邮件通知、对象存储抽象 |
| 运维 | 健康检查、结构化日志（trace_id）、Alembic 迁移、Docker Compose 部署 |

**当前已实现**（约 40% L1）：认证权限体系、文档上传解析（text + PDF）、检索骨架（关键词 + 向量 + 合并去重）。

**剩余 L1 工作**：

1. embedding 写入 Milvus 闭环

7. Word/Excel/PPT/Markdown 解析器扩展
8. 管理台完善（问答门户 + 文档管理页面完善）
9. 反馈 API + 前端
10. OIDC SSO
11. SMTP 邮件通知
